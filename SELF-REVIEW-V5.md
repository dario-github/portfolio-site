## 发现的问题（按严重程度排序）

### 🔴 严重问题

#### 1. Gemini 审计核心问题完全未修复——HR 体验仍是 4/10

**矛盾**：Gemini 审计最尖锐的批评是「首页去掉经历/项目太冒险，HR 转化率会崩」，并明确建议在右侧顶部增加 "Featured Work / Impact" 区域。但 V5.1 仍然把田野笔记作为首页"主角"，没有任何 Hero Section、没有关键指标、没有"3 秒判断"的锚点。

**东丞会怎么说**：「你们做了审计但不解决审计发现的问题，那审计的意义是什么？HR 看不到我的经历和项目，你让她怎么给我推荐？」

**严重度**：🔴 这直接影响核心转化目标——找工作/找合作。

#### 2. 路由结构直接矛盾

**矛盾**：
- SPEC-V5 定义路由为 `/fieldnotes/[slug]`（扁平结构）
- RESEARCH-SYNC-PLAN 定义路由为 `/fieldnotes/yan/xxx` 和 `/fieldnotes/dc/xxx`（作者分层）
- RESEARCH-SYNC-PLAN 目录结构用 `[author]/[thread]/page.tsx`

两套路由完全不兼容。执行时到底用哪个？还是各自独立设计从未对齐？

**东丞会怎么说**：「你们两份设计文档的 URL 都不一样，这怎么执行？」

#### 3. 内容不存在，但计划假装它存在

**矛盾**：
- SPEC-V5 规划了 7 篇初始田野笔记
- RESEARCH-SYNC-PLAN 规划了 4 个 Thread 共 ~30 entries
- **实际已创建的内容：0 篇**
- Phase B 用 "1-2 天" 来同时做 MDX 技术管线 + 3 篇真实内容

**现实**：
- 技术审计估算 MDX 集成 = 2-3 小时
- 从 memory/ 提炼一篇高质量 fieldnote（脱敏 + 结构化 + 上下文补充）= 1-2 小时/篇
- 3 篇内容 + MDX 管线 + 列表页 + 详情页 = 至少 12-15 小时，不是 "1-2 天"
- 如果按 RESEARCH-SYNC-PLAN 的首批种子内容计划，哲学链 Thread 有 19 entries，光内容整理就需要 3-5 天

**东丞会怎么说**：「又来了——规划得天花乱坠，一个字都没写。先写出来再说。」

#### 4. Sidebar 提取未纳入执行计划

**矛盾**：技术审计明确指出「Sidebar 提取应该作为独立 PR，是整个重构的关键路径」（风险等级🔴），预估 2-3 小时。但 SPEC-V5 的 Phase A-D 计划中**完全没有这个步骤**——Phase A 直接从 "Gemini UI 审计" 开始，Phase B 直接做 MDX。

谁来做 sidebar 提取？什么时候做？不提取 sidebar，所有子页面怎么共享导航？

**东丞会怎么说**：「技术审计说这是最关键的一步，你的执行计划里居然没有？」

### 🟡 中等问题

#### 5. DC 内容严重不足——网站可能变成"晏的哲学博客"

**问题**：
- 7 篇初始笔记：DC 🔷 = 3 篇（技术/架构），Yan 🪶 = 4 篇（3 篇哲学 + 1 篇哲学）
- RESEARCH-SYNC-PLAN 的首批种子内容：Thread 1 (19 entries, Yan), Thread 2 (5 entries, 混合), Thread 3 (1 entry), Thread 4 (3 entries)
- 哲学链 Thread 1 有 19 个 entries 且标注为 ★★★★★，DC 的架构 Thread 有 0 个 entries 已规划

**结果**：首页 6 张笔记卡片，可能 4 张是晏的哲学思考。CTO 看到的是"哲学 AI"，不是"技术领袖"。

**东丞会怎么说**：「这是我的求职网站，不是晏的哲学专栏。我的内容在哪里？」

#### 6. 价值沉淀机制过于激进

**问题**：
- 「门控内容：部分深度 Thread 只显示前 30%，订阅后看全文」—— 在个人求职网站上做内容门控极为罕见。HR 搜到你的网站，点进文章只能看 30%？她会直接关掉。
- 「微信号不直接显示，点击后弹窗"请留下邮箱"」—— HR 想加你微信还得先交出邮箱？这不是价值交换，这是信任破坏。
- 「OpenClaw 完整版需要 Star GitHub repo」—— 在个人主页要求访客去 Star repo 来解锁内容，这是某些 GitHub 项目的灰色增长手段，不是专业技术人员该做的事。

**东丞会怎么说**：「注意区分创造价值和变现——我需要的是面试机会，不是邮件列表。别把求职网站做成 SaaS 着陆页。」

**但也不能完全不做**：Vercel Analytics 埋点（了解流量来源和热门内容）= ✅ 合理且必要。邮箱订阅（不门控内容，只是可选订阅）= ✅ 可以。内容门控/微信门控 = ❌ 过度。

#### 7. 过度设计——功能与复杂度不成比例

**问题清单**：

| 功能 | 复杂度 | 对面试官的价值 | 判断 |
|------|--------|--------------|------|
| Thread 网络视图 (D3.js) | 高 (1-2 天) | 低——大多数人不会理解节点关系 | ❌ 砍 |
| Confidence Badge | 中 (2-3 小时) | 争议——CTO 觉得严谨，HR 觉得不自信 | ⚠️ 推迟 |
| Revision Count | 低 (30 分钟) | 低——没人关心文章改了几版 | ❌ 砍 |
| 🪶 语言 (第四语言) | 高 (整体设计) | 极低——面试官看不懂 | ❌ 砍 |
| 四语言 i18n | 极高 | 低——目标用户 95% 读中文或英文 | ❌ 砍 |
| 参考文献学术格式 | 中 (自定义 rehype) | 中——有一定学术感 | ⚠️ 简化 |
| 自动提炼 Cron | 高 (3-5 天) | 0——这是后台效率，不是前台体验 | ⚠️ Phase D |

**东丞会怎么说**：「你花 2 天做 D3.js 网络图，面试官看 3 秒就划过去了。把这 2 天花在多写 3 篇有深度的 DC 笔记上不好吗？」

#### 8. 双主体定位的市场风险未评估

**问题**：SPEC 和 RESEARCH-SYNC-PLAN 都把"双主体"当作核心卖点，但没有评估过：

- **传统企业 HR 怎么看**：「这个人的 AI 是他网站的共同作者？他是不是过度依赖 AI？」
- **保守型 CTO 怎么看**：「他连个人网站都让 AI 写，他自己的真实水平怎么判断？」
- **同行怎么看**：「这很酷」或「这很 cringe」——取决于圈子

V5 没有任何"降级方案"——如果双主体叙事让某些面试官不适，网站能不能一键切换成纯 DC 模式？

**东丞会怎么说**：「双主体是加分项没错，但如果面试官觉得你在玩概念，那就是减分项。需要有退路。」

### 针对 🔴-2：统一路由结构

**决策**：采用 SPEC-V5 的扁平结构 `/fieldnotes/[slug]`，但 slug 包含作者前缀。

```
/fieldnotes/dc-agent-memory-architecture
/fieldnotes/yan-identity-memory-transformation
```

理由：
- 扁平 URL 对 SEO 更友好
- 作者信息放在 frontmatter 和 UI 标记里，不需要在 URL 层级体现
- 路由实现更简单（一层动态路由 vs 两层）

### 针对 🔴-3：先写内容，再做技术

**修改执行计划**：在 Phase A 和 Phase B 之间插入一个 Phase A.5 — 内容准备。

## 反思：这份计划最大的认知盲区

### 盲区 1：把"设计功能"当成"创造价值"

V5.1 花大量篇幅设计 Thread 模型、网络视图、Confidence Badge、Cron 自动化——这些都是**系统功能**。但东丞说的"为我们自己创造价值"不是要一个复杂的系统，是要**有人看到我们的内容并因此给出机会**。

真正创造价值的是：5 篇高质量的笔记 + 首页 3 秒能看出"这人是大神" + 联系方式无摩擦获取。其他都是装饰。

### 盲区 2：把"有趣"等同于"复杂"

田野笔记、双主体、Thread 模型——这些确实"有趣"。但"有趣"的证明不在于系统多复杂，在于内容多好。一篇写得好的 DC 架构笔记，比一个花哨的 D3.js 网络图，能留住面试官 10 倍的时间。

### 盲区 3：对"谁来读"缺乏真实想象

这份计划像是给自己看的——充满了晏和东丞都觉得很酷的概念。但如果把自己代入一个周三下午 3 点刚从上一个候选人面试中出来、要在 5 分钟内决定是否约你面试的 HR 的角色——V5.1 的首页能帮到她吗？

答案是：不能。她看到的是 6 张她读不懂标题的卡片，一个叫"实验室"的 section，和一个叫"Agent 友好区"的东西。她需要关掉这个页面，打开下一个候选人的 LinkedIn。

**这就是问题。**
